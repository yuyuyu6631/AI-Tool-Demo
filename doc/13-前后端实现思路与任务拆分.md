# 前后端实现思路与任务拆分

## 1. 当前架构情况

当前项目已经形成稳定的 Monorepo 基线：

- `apps/web`：公开站点与后台前端
- `apps/api`：统一后端服务
- `packages/contracts`：共享契约收敛点

当前实现主线不再是“从旧 Demo 迁移”，而是“在现有主干上继续补齐闭环”。

## 2. 当前前后端职责边界

### 2.1 前端 `apps/web`

职责：

- 承接首页、详情、场景、对比、认证和后台路由
- 以 URL 参数为主要状态源承接目录逻辑
- 在客户端恢复登录态并控制后台访问

### 2.2 后端 `apps/api`

职责：

- 提供目录、评论、AI、认证、后台管理接口
- 维护数据库模型与迁移
- 提供抓取/解析入口和定时任务骨架

## 3. 当前优先任务

### Task 1：继续统一文档和自动基线

- 让 `docs/current-implementation-baseline.*` 成为稳定事实入口
- 避免正文文档继续手工漂移

### Task 2：收敛共享类型

- 减少 `apps/web` 与 `packages/contracts` 重复定义
- 明确共享包的长期收敛路径

### Task 3：补齐后台闭环

- 让工具、评论、榜单管理更完整
- 将 `tool_updates` 与审核发布串起来

### Task 4：补测试与错误态

- 扩展后台与评论页面测试
- 明确更多空态、错态、超时态

### Task 5：补运营与增长能力

- SEO / 专题能力
- 线索收集
- 更完整的权限与工作流

## 4. 当前任务优先级建议

1. 先稳定文档和共享类型
2. 再补后台闭环和测试
3. 再推进抓取审核闭环
4. 最后补增长与商业化能力

## 5. 当前不应再继续写错的口径

- 不再把首页外的 `/tools` 写成主目录承接页
- 不再把“后台管理未落地”写成当前事实
- 不再把“评论未实现”写成当前事实
- 不再把浏览器 `mock API` 写成当前方案
